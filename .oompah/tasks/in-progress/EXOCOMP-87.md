## Summary

Implement the pure PKI library that validates node Certificate Signing Requests and issues short-lived node leaf certificates from the online intermediate CA. This is a prerequisite for both the enrollment and renewal HTTP endpoints and contains no network or HTTP code.

### Context

EXOCOMP-75 delivered the coordinator PKI bootstrap in apps/exocomp_coordinator/lib/exocomp/coordinator/pki/bootstrap.ex using the x509 hex library (already a dependency). The online state directory contains intermediate_ca.pem and intermediate_ca_key.pem. EXOCOMP-77 exposed this via the PKI.State GenServer. EXOCOMP-76 delivered EnrollmentToken with inventory-bound token issuance/consumption.

The coordinator inventory entry (Exocomp.Coordinator.Inventory.Node) includes a certificate_identity field representing the expected node SAN. The root fingerprint is SHA-256 of the DER-encoded root cert, formatted as colon-separated uppercase hex pairs.

### Implementation

Create apps/exocomp_coordinator/lib/exocomp/coordinator/pki/issuer.ex containing Exocomp.Coordinator.PKI.Issuer with:

### CSR validation (validate_csr/2)

Accepts a PEM-encoded CSR binary and an expected node_id string. Validates:
- Valid PEM format and parseable X.509 CSR structure (use x509 library)
- Key type: EC with P-256 curve (secp256r1) minimum; reject RSA below 3072 bits
- Key size / strength: EC P-256 minimum
- Subject Alternative Name: must contain exactly DNS:node_id (matching the expected node_id argument); reject if SAN is absent, contains extra names, or does not match
- basicConstraints: CA:false or absent; reject any CSR with CA:true
- keyUsage in CSR attributes: must include digitalSignature; reject if keyEncipherment or keyCertSign is present
- extendedKeyUsage: must include clientAuth (for mTLS renewal); reject if serverAuth is absent and check for required extensions
- CSR signature: self-signature must verify with the enclosed public key
- Returns {:ok, parsed_csr} or {:error, Exocomp.Coordinator.Error.t()}

### Leaf cert issuance (issue_leaf/3)

Accepts a validated CSR, the online_state directory path, and keyword options (validity_days defaulting to 30). Reads intermediate_ca.pem and intermediate_ca_key.pem from the online state. Issues a short-lived X.509 leaf certificate:
- Signed by the online intermediate CA
- Subject: /CN=node_id/O=Exocomp
- SAN: DNS:node_id (from validated CSR)
- CA: false
- keyUsage: digitalSignature
- extendedKeyUsage: clientAuth, serverAuth
- Validity: now to now + validity_days (default 30 days)
- Returns {:ok, chain_pem} where chain_pem concatenates the leaf cert PEM followed by the intermediate cert PEM (not the root)
- Returns {:error, Error.t()} if intermediate key/cert cannot be loaded or signing fails
- Private keys read from disk must never be logged or included in error messages

### Tests

Create apps/exocomp_coordinator/test/exocomp/coordinator/pki/issuer_test.exs. Use disposable temporary directories with a real coordinator PKI state generated by PKI.Bootstrap.initialize in test setup. Cover:
- Valid EC P-256 CSR with correct SAN → issuance succeeds, chain verifies against root
- Invalid PEM (not a CSR) → error
- RSA key below 3072 bits → error
- SAN missing → error
- SAN mismatch (different node_id) → error
- Extra SANs in addition to correct one → error
- CA:true in basicConstraints → error
- Missing digitalSignature keyUsage → error
- Self-signature tampered → error
- Issued chain verifiable against the root cert
- Issued cert has correct 30-day validity window
- Issued cert has correct SAN
- Reject CSR from intermediate key (not a node key)

Run make fmt-check, make lint, make test before closing.
