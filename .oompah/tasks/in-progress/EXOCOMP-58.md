## Summary

### Goal
Implement \`Exocomp.Node.Config\` — the module that loads, validates, and exposes node configuration.

### Context
The Elixir scaffold is in \`apps/exocomp_node/\` (from EXOCOMP-7). Fixture config files are generated by sibling task EXOCOMP-57 and live at \`apps/exocomp_node/test/fixtures/\`. This module is a dependency for identity validation and listener startup.

### Config Schema (version 1)
\`\`\`json
{
  "version": 1,
  "node_id": "exocomp-test-node",
  "listen_address": "0.0.0.0",
  "listen_port": 4433,
  "tls": {
    "cert_path": "/path/to/node.crt",
    "key_path": "/path/to/node.key",
    "ca_path": "/path/to/ca.crt"
  },
  "diagnostic": {
    "allowed_services": ["nginx", "postgresql"]
  },
  "llama": {
    "executable_path": "/usr/local/bin/llama-server",
    "model_path": "/models/qwen.gguf",
    "model_sha256": "abc123...",
    "bind_address": "127.0.0.1",
    "bind_port": 8080
  },
  "timeouts": {
    "inference_ms": 30000,
    "diagnostic_ms": 5000,
    "task_history_limit": 100
  }
}
\`\`\`

### Implementation

### Module: \`Exocomp.Node.Config\`
File: \`apps/exocomp_node/lib/exocomp/node/config.ex\`

Public API:
- \`load(path \\ nil) :: {:ok, t()} | {:error, reason}\` — loads from \`path\` or falls back to \`EXOCOMP_CONFIG_FILE\` env, then defaults
- \`t()\` — a struct with all required fields as atoms

Validation rules:
- File must exist and be readable
- Must be valid JSON
- \`version\` must be \`1\` (reject any other value with clear error)
- All required fields must be present (return a list of missing field paths)
- Field types must match (strings, integers, lists as above)

Environment overrides (applied after file load, before validation):
- \`EXOCOMP_NODE_ID\` overrides \`node_id\`
- \`EXOCOMP_LISTEN_ADDRESS\` overrides \`listen_address\`
- \`EXOCOMP_LISTEN_PORT\` overrides \`listen_port\` (parse as integer)
- \`EXOCOMP_TLS_CERT_PATH\`, \`EXOCOMP_TLS_KEY_PATH\`, \`EXOCOMP_TLS_CA_PATH\`

### Secret redaction
Error messages and Logger output must NEVER include the value of \`key_path\` or certificate content (PEM bytes). When reporting an error about a key or cert file, use only the label (e.g., \`"tls.key_path"\`) not its value. Implement a \`Exocomp.Node.Redact\` helper or use an inline redact function.

### Dependency: Jason
Add \`{:jason, "~> 1.4"}\` to \`apps/exocomp_node/mix.exs\` deps.

### Tests
File: \`apps/exocomp_node/test/exocomp/node/config_test.exs\`

Required test cases:
- \`load/1\` with missing file returns \`{:error, :enoent}\`
- \`load/1\` with malformed JSON returns \`{:error, {:json_parse, _}}\`
- \`load/1\` with unknown version returns \`{:error, {:unsupported_version, 99}}\`
- \`load/1\` with missing required field returns \`{:error, {:missing_fields, [...]}}\`
- \`load/1\` with valid config returns \`{:ok, %Config{}}\` with all fields populated
- Environment variable \`EXOCOMP_NODE_ID\` overrides \`node_id\` in file
- Error message for key_path issues does not contain the key path value (redaction)

### Quality Gate
\`make test\` and \`make lint\` must pass.
