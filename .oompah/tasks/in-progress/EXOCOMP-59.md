## Summary

### Goal
Implement \`Exocomp.Node.Identity\` — validates that the node's TLS certificate chain is valid, the leaf cert's SAN matches the configured node ID, and the private key file has secure permissions.

### Context
The Elixir scaffold is in \`apps/exocomp_node/\` (from EXOCOMP-7). Fixture certificates are generated by EXOCOMP-57. Config loading is implemented by EXOCOMP-58. This module is called by the mTLS listener (EXOCOMP-59) before starting Bandit.

Use BEAM built-ins: \`:public_key\`, \`:ssl\`, \`:crypto\`, and \`File.stat/2\`. No third-party crypto libraries.

### Implementation

### Module: \`Exocomp.Node.Identity\`
File: \`apps/exocomp_node/lib/exocomp/node/identity.ex\`

Public API:
- \`validate(config :: Exocomp.Node.Config.t()) :: :ok | {:error, reason}\`

Validation steps (in order, fail-fast):

1. **Key file permissions**: \`File.stat!(config.tls.key_path, time: :posix)\` — check that \`stat.mode & 0o077 == 0\` (no group or world read/write/execute bits). Return \`{:error, :key_not_secure}\` if the check fails. Do NOT log or include the key path in error payloads.

2. **Parse trust root**: Read \`config.tls.ca_path\`, decode PEM to DER, parse as OTP certificate with \`:public_key.pem_decode/1\` and \`:public_key.pkix_decode_cert/2\`.

3. **Parse cert chain**: Read \`config.tls.cert_path\`, decode and parse all PEM entries.

4. **Verify chain**: Use \`:public_key.pkix_path_validation/3\` (or \`:ssl.pkix_verify_hostname/3\` as a supplement) to verify the leaf certificate is signed by the trust root. Return \`{:error, {:invalid_chain, reason}}\` without leaking cert bytes.

5. **SAN check**: Extract Subject Alternative Names from the leaf cert using \`:public_key.pkix_decode_cert/2\`. Look for a \`dNSName\` or \`uniformResourceIdentifier\` entry that matches \`config.node_id\`. Return \`{:error, {:san_mismatch, expected: config.node_id}}\` (never include the actual SANs in the error; they may reveal network topology).

### Secret redaction
Implement in \`Exocomp.Node.Redact\` (or extend from EXOCOMP-58):
- A Logger metadata filter that strips any key tagged \`:key_path\` or \`:cert_pem\`
- Any error tuple returned by this module must not contain file paths to key files or raw PEM bytes

### Logger filter
Add the redaction filter to \`config/config.exs\` or in the application start callback:
\`\`\`elixir
:logger.add_primary_filter(:redact_secrets, {&Exocomp.Node.Redact.filter/2, []})
\`\`\`

### Tests
File: \`apps/exocomp_node/test/exocomp/node/identity_test.exs\`

Required test cases (use fixture certs from EXOCOMP-57):
- Key file with group-readable mode (0o640) returns \`{:error, :key_not_secure}\`
- Key file with world-readable mode (0o644) returns \`{:error, :key_not_secure}\`
- Key file with mode 0o600 proceeds to next check (does not return key_not_secure)
- Cert signed by rogue CA returns \`{:error, {:invalid_chain, _}}\`
- Expired cert returns \`{:error, {:invalid_chain, _}}\`
- Cert with wrong SAN (mismatched node ID) returns \`{:error, {:san_mismatch, _}}\`
- Valid cert chain + SAN match + secure key returns \`:ok\`
- Error messages do not contain key file path strings (redaction)
- Logger output does not contain cert PEM bytes (use log capture in test)
